***iSCSI (от англ. Internet Small Computer System Interface — «интерфейс малых компьютерных систем интернета»)*** - стандартный протокол передачи данных. Благодаря ему SCSI-команды, применяемые для управления устройствами хранения, передаются через сети TCP/IP. Так, существующую инфраструктуру сетей TCP/IP можно использовать для доступа к удаленным блочным устройствам - жестким дискам, как если бы они были подключены локально 

***iSCSI как и NFS потерпел несколько итераций:*** 
- ISCSI v0 - предварительный вариант с базовыми функциями для передачи SCSI-команд через TCP/IP, появившийся в конце 1990-х
- ISCSI v1 - официально утвержденная версия, которую определили RFC 3720 в 2004 году. Она включала ряд улучшений: поддержку CHAP для аутентификации, оптимизацию производительности 
- ISCSI v2 - определена в RFC 5048 в 2007 году. Эта версия не изменила протокол существенно, но внесла некоторые уточнения и корректировки
- ISCSI v3 - определена в RFC 7143 в 2014 году с рядом значительных улучшений: поддержкой поиска и восстановления ошибок (англ. Error Recovery Level), качества обслуживания (англ. Quality of Service, или QoS) и других опций
- ISCSI v4 - последнее обновление iSCSI, случившееся в январе 2022 года. Стандарт iSCSI v4 ещё не утверждён окончательно, но предполагается, что он ещё улучшит производительность, безопасность и возможности протокола. К примеру, благодаря поддержке более высоких скоростей передачи данных и интеграции с современными технологиями хранения вроде NVMe (от англ. Non-Volatile Memory Express)
- Инициация соединения
	- Инициатор (клиентское устройство) начинает соединение с целью (устройством хранения данных) по сети TCP/IP. Обычно это происходит через указанный на целевом устройстве порт TCP 3260 
	- Инициатор определяет цель с помощью IP-адреса или DNS-имени и может указать порт для соединения 
- Аутентификация (опциональный шаг)
	- Установив соединение, инициатор и цель могут провести аутентификацию для безопасной передачи данных. Один из наиболее распространенных методов - CHAP 
- Сеанс и обмен параметрами
	- Дальше инициатор и цель устанавливают сеанс и обмениваются параметрами: определяют доступные блочные устройства, протокол обмена данными, например TCP или UDP, максимальный размер передаваемых блоков и так далее
- Передача SCSI-команд
	- Инициатор отправляет цели SCSI-команды — это может быть чтение, запись, запрос состояния — через установленное TCP-соединение
	- Цель получает команды и затем выполняет соответствующие им операции с данными на своих устройствах хранения
- Передача данных
	- Цель может возвращать результаты выполненных команд и данные обратно инициатору через то же TCP-соединение
	- Также обмен информацией может включать пересылку больших блоков информации в обоих направлениях
- Завершение соединения
	- Когда передача завершена или это просто необходимо, TCP-соединение между инициатором и целью разрывается

***iSCSI состоит из нескольких основных компонентов - еще их называют сервисами, и каждый из них выполняет определённые функции в процессе передачи информации:*** 
1. ***Инициатор iSCSI (iSCSI Initiator)*** - программное или аппаратное устройство,  инициирующее запросы к удаленному устройству хранения данных по протоколу iSCSI. Оно обеспечивает интерфейс для отправки SCSI-команд цели и обработки ответов от нее 

***Основные службы, которые запускаются при работе этого компонента в Linux:*** 
- **iSCSI Initiator Daemon** (`iscsid`) - основной демон, который управляет установлением соединения с удаленными iSCSI-устройствами 
- **iSCSI Discovery Daemon** (`iscsidtd`) - отвечает за поиск iSCSI-устройств и добавление их в список известных целей. Он ищет доступные устройства по сети по подключает их к системе 
- **iSCSI Multipath** - независимой компонент, с которым можно использовать несколько путей для подключения к одной цели iSCSI. Он балансирует нагрузку и обеспечивает отказоустойчивость при использовании iSCSI-устройств 
- **iSCSI Service** - отвечает за общий контроль и управления передачей данных между инициатором и удаленным iSCSI-устройством. Эта служба работает в тесном взаимодействии с iscsid для стабильного и надежного соединения
- **TCP/IP Networking** - стандартный протокол передачи данных через IP-сеть в Linux TCP/IP. Для работы iSCSI Initiator нужен установленный и правильно настроенный TCP/IP
- **Multipathd** - отвечает за мониторинг и управление многопутевыми соединениями, созданными с помощью iSCSI Multipath. Он обеспечивает автоматическое восстановление соединения в случае отказа одного из путей

2. ***Цель iSCSI (iSCSI Target)*** - устройство, предоставляющее доступ к блочным устройствам хранения данных по протоколу iSCSI

*Цель обрабатывает полученные от инициатора SCSI-команды, выполняет операции чтения и записи на блочных устройствах и возвращает результаты обратно инициатору*

- **iSCSI Target Daemon** (`iscsid`) - основная служба, ответственная за обработку запросов от инициаторов iSCSI и контроль удалённых блочных устройств. Обычно запускается автоматически при загрузке системы и управляется менеджером служб — в большинстве современных дистрибутивов Linux это `systemd`

*Когда iSCSI-инициатор подключается к iSCSI-цели, iscsid обрабатывает запрос, аутентифицирует инициатора (если этот шаг настроен) и предоставляет доступ к запрошенным блочным устройствам* 

- **iSCSI Target Management Utility** (`tgtadm`) - утилита, с помощью которой администратор может управлять iSCSI Target и его конфигурацией из командной строки
- **iSCSI Target Portal Group Daemon** (`iscsiportald`) - отвечает за управление сетевыми порталами iSCSI: отслеживает доступные и обрабатывает подключения инициаторов к правильным блочным устройствам

***Портал iSCSI*** - комбинация IP-адреса и TCP-порта, через которые iSCSI-инициатор подключается к цели iSCSI. Порталы используются для идентификации и доступа к целям ISCSI через TCP/IP 

- **iSCSI Target Authentication Daemon** (`iscsi_trgt`) - дополнительная служба для аутентификации и авторизации инициаторов iSCSI. Она тоже запускается автоматически вместе с системой и управляется менеджером служб

3. Механизм аутентификации - опциональный компонент, который обеспечивает аутентификацию между инициатором и целью для безопасной передачи данных. Методы в этом случае могут быть разными, включая CHAP

## ***Полезные команды и утилиты***

`sudo dd if=/dev/zero of=/srv/shares/iscsi/lun.img bs=1M count=4096`

 - `dd` - в Linux используется для копирования данных, создания файлов или дискового образа
 - `/dev/zero` - специальное устройство, генерирующее поток нулей при чтении. `if` сообщает `dd` источник данных для копирования
 - `of=/storage/lun.img` - опция `of` указывает `dd` место, куда запишется результат копирования. В нашем случае создаётся файл `lun.img` в директории `/storage`
 - `bs=1M` - определяет размер блока данных, который `dd` будет копировать за один раз. `1M` — 1 Мб
 - `count=4096` - передаёт `dd`, сколько блоков данных нужно скопировать. У нас их 4096, что при размере блока в 1 Мб даёт файл в 4 Гб

***LUN (Logical Unit Number — «номер логической ячейки»)*** - это виртуальное устройство, выступающее в качестве цели iSCSI для доступа к дисковому пространству. Оно может быть представлено как файл, блочное устройство или RAID-массив 

*LUN с номером `0` обычно зарезервирован для описания самой цели, поэтому у нас его значение — `1`*

`iscsiadm -m session -P3 -` обратиться к логическому блоку 

- `iscsiadm` - сама команда для управления iSCSI-устройствами в Linux
- `m session` - задаёт режим работы команды, в нашем случае — `session`, что означает работу с текущими сеансами iSCSI
- `P3` - указывает на уровень подробности вывода (verbosity level). `3` обозначает её наибольший уровень — команда будет выводить максимальное количество информации о текущих iSCSI-сессиях

***Вопрос безопасности:*** 
1. Аутентификация 
***CHAP** используется в iSCSI для проверки подлинности инициатора и цели. В его основе — принцип «вызов — ответ», где сервер отправляет вызов аутентификации инициатору. Последний должен ответить, используя предварительно согласованные учётные данные.*

*Всё это помогает предотвратить атаки вроде подслушивания и защитить учётные данные от несанкционированного доступа.*

***Mutual CHAP** — расширение CHAP, которое обеспечивает двустороннюю аутентификацию. Это означает, что инициатор и цель аутентифицируют друг друга для предотвращения атак с подменой и подслушиванием*

2. Шифрование 
***IPsec** (англ. Internet Protocol Security) можно использовать для шифрования iSCSI-трафика: работая на уровне IP, он пропускает трафик через зашифрованный туннель между инициатором и целью. IPsec предоставляет механизмы для шифрования (как симметричного, так и асимметричного) и аутентификации (через разные методы вроде HMAC).*

***IKE** (англ. Internet Key Exchange) — протокол установки ключей, используемый в IPsec для управления процессом аутентификации и обмена ключами. Он помогает инициатору и цели согласовать общие параметры безопасности и установить надёжный канал для передачи данных.*
