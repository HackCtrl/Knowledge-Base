***PostgreSQL*** - современная СУБД с открытым исходным кодом и свободной лицензией 
## ***Установка***

***Обновим систему:*** 
- `sudo apt update` 
- `sudo apt upgrade` 

***Установим PostgreSQL:*** 
- `sudo apt install postgresql`

***Проверим установку - автозапуск и статус служб:*** 
- `systemctl is-enabled postgresql`
- `systemctl status postgresql`

***Если не запущен:*** 
- `sudo systemctl start postgresql`

***У PostgreSQL серверная архитектура*** - сервер управляет фалами БД, обслуживает соединение от клиентов и выполняет операции с БД, которые запрашивают клиенты. За серверную часть отвечает приложение postgres

***Клиент*** - это фронтенд-приложение, через которое пользователь выполняет операции над БД. Клиентские приложения могут быть разными: от консольных утилит до веб-приложений 

***При установки из исходных кодов можно самостоятельно создать любого пользователя для работы СУБД:*** 
- `sudo su - postgres` 

![[Pasted image 20260131114148.png]]

***Теперь создадим новую роль. Для этого используется команда*** `createuser`: 
- `createuser sysadmin` 

***Удалить роль можно при помощи*** - `dropuser`

***У роли есть атрибуты, которые определяют ее полномочия:*** 

- `LOGIN` - дает право на подключение к БД 
- `CREATEDB` - на создание БД 
- `CREATEROLE` - на создание ролей 

*Атрибуты создаются при создании роли, а изменить их можно с помощью SQL-запроса* `ALTER ROLE`

***На данном этапе у нас есть сервер, БД и пользователь. Осталось настроить работу по сети***

***Чтобы к серверу можно было подключаться по сети, нужно внести правки в файл `pg_hba.conf` (`/etc/postgresql/14/main/pg_hba.conf`) — в нём описываются параметры аутентификации. Добавим в самый конец строку:*** 

- `host test_db sysadmin 10.10.0.0/16 scram-sha-256`

![[Pasted image 20260131121725.png]]

`host` - означает подключение по сети к БД `test_db` для пользователя sysadmin из сети `192.168.175.0/24` с авторизацией по паролю, зашифрованному методом `scram-sha-256`

***Проверим подключение:*** 

*Как вы уже знаете, чтобы PostgreSQL начал обрабатывать запросы из сети, нужно отредактировать файл `postgresql.conf` (`/etc/postgresql/14/main/postgresql.conf`). Найти такую строку — `#listen_addresses = 'localhost'` — и заменить её на `listen_addresses = '*'`*

![[Pasted image 20260131123915.png]]

*Отсутствие символа комментария `#`. `*` означает «слушать на любом адресе сервера, где установлен PostgreSQL», но можно указать и конкретный IP-адрес*

***Чтобы параметры применились, нужно перезапустить сервер:*** 
- `sudo systemctl restart postgresql`

***И проверяем подключение:*** 
- `psql --host 10.10.0.137 --username sysadmin --password --dbname test_db`

## ***Кластер***

*Установленный нами север может обслуживать несколько БД. Одна может быть для веб-сайта, другая - для мониторинга, и так далее*

***Обычно серверы объединяют в кластеры*** - это когда несколько серверов обслуживают одинаковый набор БД. Так повышается отказоустойчивость и производительность 

*Когда на один сервер поступает запрос на изменение данных, эти данные должны быть быстро переданы на все серверы в кластере, чтобы последующие запросы на чтение возвращали корректную информацию.*

*Но что, если на два сервера поступит запрос на изменение одних и тех же данных? Возникает проблема синхронизации*

***У этой проблемы есть разные решения, поэтому и видов кластеров существует несколько:***

- Где запросы на изменение разрешено обрабатывать только одному серверу, о остальные занимаются запросами только на чтение 
- В которых операция записи считается выполненной, только когда все серверы подтвердили получение своих копий низменных данных 
- В котором все серверы работают с одной копией БД, хранящейся на общем сетевом хранилище

*PostgreSQL поддерживает несколько видов кластеризации, мы же рассмотрим один из распространённых вариантов: трансляцию журналов на резервные серверы. В этой схеме главный сервер постоянно транслирует изменения на резервные. Если он выйдет из строя, его функции заберёт один из запасных*
## ***НАСТРОЙКА РЕПЛИКАЦИИ***

### ***Подготовка главного сервера к репликации***

***Создадим специального пользователя для репликации:*** 
- `sudo -u postgres createuser --replication -P syncuser`

![[Pasted image 20260131125705.png]]

***Проверим создание:*** 
- `sudo -u postgres psql -c "\du"`
### ***Настройка конфигурации репликации***

***Редактируем основной конфигурационный файл:*** 
- `sudo nano /etc/postgresql/15/main/postgresql.conf`
	- `wal_level = replica`
	- `max_wal_senders = 10`
	- `wal_keep_size = 1GB`
	- `listen_addresses = '*'`

***WAL (Write Ahead Log)*** - это журнал перезаписи, который обеспечивает целостность данных: информация записывается сначала в него, а затем изменяется уже в самой БД на диске.
WAL можно воспринимать как журнал транзакций, а `wal_level` определяет объем записываемых в него данных 

***Настроим доступ для репликации:*** 
- `sudo nano /etc/postgresql/15/main/pg_hba.conf`

![[Pasted image 20260131131610.png]]

***Перезапустим сервер:*** 
- `sudo systemctl restart postgresql`

***Проверим настройки:***  
- `sudo -u postgres psql -c "SHOW wal_level;"`

![[Pasted image 20260131131732.png]]

***Репликация*** - это процесс синхронизации содержимого между копиями 

*Для репликации важно заводить отдельные роли и разрешать подключение только с определенных серверов, так как БД почти всегда хранят значимую информацию* 
### ***Подготовка резервного сервера***

***Останавливаем PostgreSQL:***
- `sudo systemctl stop postgresql`

***Удаляем текущую базу данных:*** 
- `sudo rm -rf /var/lib/postgresql/15/main/*`

***Копируем базу данных с главного сервера:*** 
- `sudo -u postgres pg_basebackup -h 192.168.1.50 -D /var/lib/postgresql/15/main -U syncuser -P -v -R`
