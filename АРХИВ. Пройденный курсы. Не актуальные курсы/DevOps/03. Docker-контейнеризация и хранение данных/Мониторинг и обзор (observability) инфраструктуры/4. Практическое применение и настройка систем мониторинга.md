
***Zabbix, Prometheus, Grafana*** - прекрасно себя чувствуют в контейнерах. Их можно развернуть как привычным путем - `скачивание → установка → настройка → запуск`, так и с помощью подготовленных контейнеров, которые нужно адаптировать под свою систему  

***Разница двух методов:*** 

***Установка из пакетов:*** 
1. Производительность 
2. Легкость управления 
3. Доступ к ресурсам 

***Запуск из контейнера:*** 
1. Изоляция 
2. Портативность 
3. Интеграция 

***Что выбрать ?***
- Если нужна гибкость и портативность, то выбираем **контейнеры**, они предпочтительны для развертывания систем мониторинга в распределенной среде 
- Если важна производительность и надежность доступа к ресурсам хоста, то подойдет **Установка из пакетов** 
## ***Установка Zabbix из пакетов***

***Zabbix*** - это система с централизированной архитектурой. Она состоит из нескольких компонентов, каждый из которых выполняет свою функцию в процессе мониторинга и управления IT-инфраструктурой. Чтобы начать работу, нужно установить как минимум сервер, веб-интерфейс и агент. Установка из пакетов - один из самых распространённых способов развертывания Zabbix

***В контексте Zabbix сервер*** - центральный компонент, который собирает и обрабатывает данные, полученные от агентов или через SNMP, IPMI и другие протоколы 
#### ***1. Zabbix Server***

Это ядро всей системы. Сервер обрабатывает метрики, сохраняет из в БД, отправляет уведомления, генерирует отчеты и отображает данные во встроенном веб-интерфейсе. Именно он координирует все взаимодействие между остальными компонентами системы 
#### ***2. Zabbix Agent***

Агент устанавливается на целевых хостах и собирает данные о состоянии ОС и запущенных сервисов. Например, о загрузке процессов, использования памяти, статусе приложений. Эти данные отправляются на сервер Zabbix для анализа 

***Zabbix-агент может работать в двух режимах:*** 
- ***Активном*** - сам отправляет данные на сервер 
- ***Пассивном*** - ожидает запросов от сервера и передает данные по требованию 
#### ***3. База данных***

Zabbix требует СУБД для хранения конфигурации, событий, истории метрик и других данных. Обычно использует PostgreSQL или MySQL/MariaDB. На этом этапе нужно создать БД, пользователя и передать параметры подключения в конфигурацию сервера 
#### ***4. Веб-интерфейс***

Для визуализации данных используется веб-интерфейс на базе PHP. Через него можно просматривать графики,  настраивать хосты и триггеры, создавать шаблоны и управлять пользователями. Интерфейс работает через веб-сервер - Apache или Nginx
#### ***Пример установки Zabbix Server (Ubuntu 22.04)**:*

```
# Добавим репозиторий Zabbix
wget https://repo.zabbix.com/zabbix/6.4/ubuntu/pool/main/z/zabbix-release/zabbix-release_6.4-1+ubuntu22.04_all.deb
sudo dpkg -i zabbix-release_6.4-1+ubuntu22.04_all.deb
sudo apt update

# Установим сервер, веб-интерфейс и агент
sudo apt install zabbix-server-mysql zabbix-frontend-php zabbix-apache-conf zabbix-agent 
```

***После установки потребуется:***
- настроить подключение к базе данных в конфигурации сервера (`/etc/zabbix/zabbix_server.conf`);
- инициализировать базу данных: создать схему и импортировать начальные данные;
- запустить сервер и агент с помощью systemd.
#### ***Компоненты Zabbix в действии***

1. ***Сбор метрик***
- Агент или другие механизмы (например: SNMP) собирают метрики и отправляют их на сервер 
2. ***Хранение данных*** 
- Все данные сохраняются в базу - как история метрик так и события связанные с триггерами 
3. ***Оповещение*** 
- При срабатывании триггера (например: превышенная загрузка CPU) Zabbix может отправить уведомление на почту или в мессенджер 
4. ***Визуализация*** 
- Веб-интерфейс показывает графики, дашборды, списки событий и алертов в режиме реального времени 

***Примеры использования компонентов Zabbix:***

- ***Zabbix Agent*** - мониторинг серверов, рабочих станций, служб и процессов в ОС 
- ***Zabbix Proxy*** - помогает собирать данные из распределенных локаций (например: в филиалах), передавая их на центральный сервер 
- ***SNMP-мониторинг*** - наблюдение за сетевым оборудованием (маршрутизаторы, принтеры, сканеры)
- ***IPMI*** - отслеживание аппаратного состояние серверов (температура, напряжение, кулер и т.д)
## ***Установка Prometheus из пакетов***

***Prometheus*** - обходится без агентов на монтируемых хостах - вместо них он использует экспортеров, для сбора метрик и мониторинга приложений систем 

***В контексте Prometheus экспортёры*** - программные компоненты, предоставляющие доступ к метрикам и статистике приложений, сервисов или устройств 

***Компоненты работы Prometheus:***

1. ***Экспортеры*** - программы, которые представляют доступ к внутренним метрикам и метаданным приложений и систем. Они работают как мост между приложением и Prometheus, экспортируя данные в понятном для него формате 
2. ***Протокол HTTP*** - Prometheus собирает метрики от экспортеров именно по этому протоколу: обычно данные представляются в текстовом формате Exposition, который Prometheus умеет читать и обрабатывать 
3. ***Система сбора и хранения данных*** - периодически Prometheus делает запросы к экспортерам для получения метрик и сохраняет их в собственном временном родовом хранилище, обеспечивая возможность визуализации и анализа данных 

***Работа экспортеров:*** 

1. ***Сбор метрик*** - Экспортеры работают внутри приложений или сетевых устройств и собирают метрики - например, по использованию ресурсов (CPU, память), количеству запросов, задержки и другим показателям, которые важны для мониторинга и анализа работы приложения 
2. ***Предоставление метрик*** - Также они предоставляют собранные метрики в формате, соответствующем формату сбора данных Prometheus - как правило, это Exposition
3. ***HTTP Интерфейс*** - Обычно экспортеры предоставляют HTTP-интерфейс, по которому Prometheus делает запросы для получения актуальных данных  

***Примеры приложений экспортёров:***

1. ***Node Exporter*** - один из наиболее популярных экспортеров для мониторинга системных ресурсов хоста 
2. ***MySQL Exporter*** - собирает метрики о работе MySQL-сервера
3. ***Blackbox Exporter*** - проверяет доступность и производительность сетевых конечных точек
