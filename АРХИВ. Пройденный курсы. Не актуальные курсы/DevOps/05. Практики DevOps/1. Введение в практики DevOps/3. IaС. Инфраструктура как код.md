## ***Что такое IaC и когда он нужен***

***IaC (от англ. Infrastructure as Code)*** - это подход с наборами инструментов для автоматизации администрирования, а так же развертывания, масштабирования, настройки и обновления инфраструктуры с помощью кода 

***Основные составляющие laC:*** 
- Система для хранения информации и запуска сценариев 
- Система развертывания или деплоя 
- Системы управления конфигурацией

***Дополнительные составляющие:*** 
- Система для хранение секретов (логинов и паролей) и динамических данных конфигурации 
- Система для обнаружения и управления сервисами 

***Сервис*** - это некая программа, которая должна быть установлена на сервере 

- Системы автоматизированного развертывания в Kubernetes
- Системы мониторинга 
- Репозитории где хранятся пакеты для Linux-систем 

***Где нужно применять IaC:*** 
- Инфраструктура часто меняется 
- Большое количество узлов и сервисов 
- Инфраструктура должна быть унифицированной, с шаблонным видом 
- Многочисленная команда 

***Несколько типов IaC:*** 
- ***Декларативный*** - С помощью конфигурационных файлов мы описываем желаемое состояние, к которому будет стремиться инструмент. То есть как бы говорим программе: «Я хочу, чтобы это было вот так». При таком типе IaC используют инструменты развёртывания (например, Hashicorp Terraform), в которых можно управлять системой виртуализации через описание ВМ или контейнеров, которые нужно создать, изменить или удалить.
- ***Императивный*** - В этом случае мы описываем скриптами шаги для достижения какой-то цели (к примеру, установки программ на машины), которые будут выполнены инструментом. Для этого сообщаем программе: «Чтобы получилось то-то, надо делать так-то». Этот тип подразумевает использование системы управления конфигурацией вроде Ansible, Puppet или Chief.
- ***Гибридный*** - Объединяет декларативный и императивный типы. Например, при одновременном использовании Terraform и Ansible мы получаем гибридный тип.

***Когда запускаются задачи из пайплайна, все конфигурационные файлы, плейбуки и роли клонируются на машину с Terraform и Ansible. После клонирования — в обобщённом виде — есть три пути:***
1. Вызывается только Terraform для развёртывания ВМ
2. Для развёртывания ВМ вызывается Terraform, которая затем вызывает Ansible-скрипты для их настройки и локальных серверов
3. Вызывается только Ansible для настроек ВМ в облаке или системе виртуализации и локальных серверов

- Git-репозиторий — более стабильная сущность, на которую выделятся больше ресурсов системы. Поэтому с ним достигается бóльшая стабильность работы, чем просто c машиной с парой сервисов, проще добиться и отказоустойчивости. Так, в случае выхода из строя машины с Terraform и Ansible инженеру не надо переживать о потерянных данных или восстанавливать их.
- PipeLine — жизненно необходимая вещь в IaC. К примеру, настройка сервисов с помощью Ansible разбивается на разные роли и плейбуки для удобства, правильности работы и читаемости. Запускать по очереди десять плейбуков не очень удобно, к тому же часто это нужно делать в правильной последовательности, и такую работу лучше оставить пайплайну и периодически дорабатывать его.
## ***Облака, API, Ansible и Terraform***

***Облака*** - это удаленные площадки, на которых находятся серверы и сетевые устройства 

***Плюсы использование облаков:*** 
- Меньшая ответственность за оборудование 
- Экономия 
- Простое управление 
- Упрощенное разворачивание кластеров серверов 
- Неограниченные ресурсы

***Недостатки:*** 
- Работоспособность и приватность данных 
- При сбоях сервисы будут не доступны

## ***Ansible***

***Ansible*** - это система управления конфигурацией, которая применяется для автоматизированной настройки сервисов, узлов сети и инфраструктуры. Она работает по `ssh` и не использует своих клиентов на узлах для их настройки 

Так же Ansible относится к императивному виду laC, поскольку нам нужно описывать задачи, которые выполняются на удаленном узле в формате YML 

***Императивный вид*** - это парадигма, при которой программа содержит прямые указания компьютеру, что должен сделать компьютер и в каком порядке должен выполнять инструкцию 

```
- name: Adding Ubuntu local repo
  block:
    - name: Adding Ubuntu default repo
      ansible.builtin.apt_repository:
        repo: "deb {{ lin_repo }}/ubuntu/ {{ ansible_distribution_release }} main restricted universe multiverse"state: present

    - name: Adding Ubuntu updates repo
      ansible.builtin.apt_repository:
        repo: "deb {{ lin_repo }}/ubuntu/ {{ ansible_distribution_release }}-updates main restricted universe multiverse"state: present

  when: ansible_distribution == "Ubuntu"
  tags:
    - configure_system
```

***Это задачи для добавления репозиториев Ubuntu с помощью `magic vars`. У задач есть обязательные компоненты:***

- параметр `name`, который обозначает название задачи;
- модуль действия вроде `ansible.builtin.apt_repository`, который включает само действие и — при необходимости — его параметры.

***Почему важен именно этот инструмент ?*** 
- Легок в использование 
- С простым форматом разметки 
- С огромным количеством дополнительных коллекций для автоматизации задач системного администратора 
- Написание на Python, дает право писать собственные коллекции для автоматизации 
- Легко интегрируется с другими сервисами, которые используют API 
- Полностью бесплатен и с отличной документацией, лидирует среди подобных систем 
## ***Terraform***

***Terraform*** - Это система декларативного типа IaC, которая используется для автоматизации развертывания, модификации, удаления и создания резервных копий виртуальных машин, контейнеров или других сущностей в облачных инфраструктурах или системах виртуализации 

***Декларативного типа IaC*** - способ описания инфраструктуры, при котором пользователь определяет желаемое состояние системы, а не последовательность шагов для его достижения. Инструмент IaC автоматически рассчитывает, какие действия необходимо выполнить, чтобы достичь указанного состояния

***Пример описания желаемой виртуальной машины в системе виртуализации VMWare:*** 
```
data "vsphere_datacenter" "dc" {
  name = "${var.vsphere_datacenter}"
}

data "vsphere_datastore" "datastore" {
  name          = "${var.vsphere_datastore}"datacenter_id = "${data.vsphere_datacenter.dc.id}"
}

data "vsphere_resource_pool" "pool" {
  for_each      = var.vsphere_virtual_machines_list
  name          = each.value["pool"]
  datacenter_id = "${data.vsphere_datacenter.dc.id}"
}

data "vsphere_network" "network" {
  for_each      = var.vsphere_network
  name          = each.value
  datacenter_id = "${data.vsphere_datacenter.dc.id}"
}

data "vsphere_virtual_machine" "template" {
  for_each      = var.vsphere_virtual_machines_list
  name          = each.value["template"]
  datacenter_id = "${data.vsphere_datacenter.dc.id}"
}

resource "vsphere_virtual_machine" "cloned_vm" {
  for_each       = var.vsphere_virtual_machines_list
  name             = each.value["name"]
  resource_pool_id = data.vsphere_resource_pool.pool[each.key].id
  datastore_id     = "${data.vsphere_datastore.datastore.id}"guest_id = data.vsphere_virtual_machine.template[each.key].guest_id
  scsi_type = data.vsphere_virtual_machine.template[each.key].scsi_type

  dynamic "network_interface" {
  for_each = {for idx, name in each.value.network: idx=>name}
    content {
      network_id = data.vsphere_network.network[network_interface.value].id
    }
  }

  dynamic  "disk" {
  for_each = {for idx, size in each.value.disks: idx=>size}
    content {
      label            = "disk${disk.key+1}"unit_number      = disk.key
      size             = disk.value
    }
  }

  num_cpus = each.value["cpu"]
  memory   = each.value.ram

  wait_for_guest_net_timeout = 0

  clone {
     template_uuid = data.vsphere_virtual_machine.template[each.key].id
  }
}
```

***В этом коде описано желаемое состояние виртуальной машины, к которому будет стремиться Terraform. Среди ключевых преимуществ инструмента можно выделить:***

- гибок за счёт огромного количества провайдеров;
- бесплатен и c отличной документацией, лидирует среди подобных систем;
- использует API-системы, что заметно упрощает использование;
- интегрируется с другими системами вроде уже упомянутой Vault.
