***В нереляционных БД (NoSQL)*** - не используется традиционная табличная структура для хранения. Они предназначены для данных, которые не соответствуют жесткой схеме или структуре 

***Особенности NoSQL:*** 
- Масштабируемость 
- Неструктурированные данные 
- Распределенные системы 

***Различия между реляционными и не реляционными БД:*** 
- ***Структура*** - у реляционных БД - жестка схема, которая определяет структуру данных, в то время как у нереляционных БД не требуют схемы и могут хранить информацию в разных форматах
- ***Организация*** - Реляционные БД организуют данные в таблицы и строки,  связанные через внешние ключи, а Нереляционные БД обычно хранят информацию в документах, ключах-значениях, графах или столбцах
- ***Транзакции*** -реляционные БД поддерживают ряд операций, известных как транзакции, которые обеспечивают гарантию согласованности т=данных 

***Транзакции*** - это логические единицы работы, в которых выполняются операции с данными. Если все в порядке, их состояние сохраняется, а в случае проблем отменяется 

- ***Масштабируемость*** -  реляционные БД обычно масштабируются вертикально, через добавление более мощного оборудования, в то время как Нереляционные БД  предназначены для горизонтального масштабирования - расширения группы сервером, работающих с не реляционными БД 

*Реляционная база данных похожа на библиотеку, где книги организованы по категориям или авторам. У каждой есть уникальный идентификатор, она может быть связана с другими книгами через общие темы или авторов.*

*Нереляционная БД похожа на склад, где товары хранятся в разных местах без чёткой организации по категориям или структуре. У каждого предмета есть уникальный идентификатор (например, штрихкод), но связи между товарами менее определены.*

***Реляционные базы подходят для таких типов данных, которым важно обеспечить целостность и согласованность:*** 
1. Системы управления заказами 
2. Учет финансовых операций 

***Нереляционные базы используются, когда у данных нет четкой структуры или когда приоритеты производительность и масштабируемость:*** 
1. Веб-аналитика
2. Хранение больших объемов данных IoT

## ***MongoDB: обзор***

*MongoDB востребована благодаря своей гибкости, масштабируемости и простоте использования. Она позволяет хранить данные в формате JSON-подобных документов, легко масштабируется для растущих объемов информации, обладает интуитивно-понятным языком запросов и множеством инструментов для работы*

***Другие особенности:*** 

- ***Агрегация данные*** - позволяет объединять информацию из нескольких документов и выполнять операции над ней. Для этого используется оператор конвейера агрегации: 
	- `$match` фильтрует документы на основе заданных критериев 
	- `$group` группирует документы по общему полю и выполняет операции над группами (например, подсчет, суммирование и так далее)
	- `$project` выбирает и преобразует поля, которые будут включены в вывод 
	- `$sort` сортирует документы на основе заданных полей 
	- `$limit` ограничивает количество возвращаемых документов 

- ***Полнотекстовый поиск*** - для поиска по тексту, который хранится в документах. Индекс полнотекстового поиска создается на полях, содержащих текст, и позволяет выполнять быстрые и эффективные поисковые запросы 
- ***Управление пользователями и ролями*** - для контроля доступа к БД и коллекциям. Так, доступом можно управлять на уровне коллекции документа и поля 
- ***Масштабируемость и распределенность*** - MongoDB может быть развернута в распределенной среде с помощью кластеров. Распределяя данные по нескольким серверам, они обеспечивают масштабируемость и отказоустойчивость 

***MongoDB поддерживает несколько архитектур кластеров:***

***Репликационный набор*** - это группа из нескольких серверов MongoDB, которые содержат копию одного и того же набора данных. Один узел назначается первичным, а остальные - вторичными. Такая архитектура обеспечивает высокую доступность и отказоустойчивость благодаря репликации данных на нескольких серверах 

***Шардированные кластеры*** - группа из нескольких серверов MongoDB, которые совместно хранят разделенные на несколько фрагментов данные, или шарды. У каждого сервера — подмножество шардов. Запросы к шардированному кластеру направляются на сервер на основе ключа распределения, который определяет, в каком шарде хранятся данные. Путём такого разделения информации обеспечивается горизонтальная масштабируемость.

***Глобальные кластеры*** - это группа их нескольких репликационных наборов, развернутых в разных географических регионах. Данные реплицируются между репликационными наборами, обеспечивая географическую распределенность и сокращая задержку для пользователей
