## ***Снепшоты*** 

***Снепшоты (snapshot) в LVM*** - временные копии томов, которые можно использовать для восстановления информации. Так, снепшоты позволяют делать резервные копии или точки восстановления перед важными  операциями: пока приложения работают с данными в исходном томе, снепшот сохраняет их состояние на момент создания снимка 

***Команда для создания снепшота в LVM из консоли:***  
- `sudo lvcreate -L 1G -s -n snap /dev/data/lv`
![[Pasted image 20260106110225.png]]

***Чтобы восстановить данные, снепшот нужно смонтировать в отдельную точку монтирования, например так:*** 
- `sudo mkdir /mnt/snapshot` 
- `sudo mount /dev/data/snap /mnt/snapshot`
- `ls /mnt/snapshot`
![[Pasted image 20260106110525.png]]

***Если же записать изменения в исходный том, в снимке они останутся неизменными. Когда снепшот больше не нужен, его можно отмонтировать и удалить следующими командами:***
- `sudo umount /mnt/snapshot`
- `sudo lvremove /dev/data/snap` 
![[Pasted image 20260106110726.png]]

*Снепшоты не заменяют собой полноценные резервные копии и регулярное резервное копирование* 
## ***Зеркалирование (mirroring)***

***RAID в LVM — RAID 0, RAID 1 и RAID 5*** - позволяет объединить несколько дисков в массив, что повышает отказоустойчивость и производительность 

`lvremove, vgremove, pvremove` - с помощью данных команд можно удалять созданные ранее логические тома, группы томов и физические тома 

- `sudo umount /mnt/data` 
- `sudo lvremove /dev/data/lv`
- `sudo vgremove data`
- `sudo pvremove /dev/vdb1`

***Процесс создания зеркала из двух разделов:*** 
1. Создаем физический том: 
- `sudo pvcreate /dev/vdb1 /dev/vdb2`

2. Создадим группу томов:
- `sudo vgcreate data /dev/vdb1 /dev/vdb2`

3. Дело за логическим томом с рейдом — RAID 1, или «зеркалом» файловой системы:
- `sudo lvcreate -L 5G -m1 -n mirror data` 
- `sudo mkfs.ext4 /dev/data/mirror`
- `sudo mount /dev/data/mirror /mnt/data`

***Через команду `lvs` можно вывести расширенную информацию о состоянии зеркала:***
- `sudo lvs -a -o +devices`
## ***Перенос данных между физическими дисками***

*В LVM можно перемещать данные между физическими дисками без простоя системы. Типичный сценарий: добавить новый диск к группе томов и переместить информацию на него. Для это используем команды - `pvmove` и `vgreduce`*

`lvremove, vgremove, pvremove` - с помощью данных команд можно удалить логические тома, группы томов и физические тома: 
- `sudo umount /mnt/data` 
- `sudo lvremove /dev/data/mirror`
- `sudo vgremove data`
- `sudo pvremove /dev/vdb1 /dev/vdb2`

***Для создания нового физического тома на дополнительном диске, нужно его добавить в группу томов:*** 
- `sudo pvcreate /dev/vdb1` 
- `sudo vgcreate data /dev/vdb1` 

***Порядок команд для создания логического тома и ФС на нем:*** 
- `sudo lvcreate -L 5G -n lv data` 
- `sudo mkfs.ext4 /dev/data/lv`
- `sudo mount /dev/data/lv /mnt/data`
- `sudo touch /mnt/data/tesfile`

`vgextend` - команда добавляет несколько физических томов в группу томов, и тем самым расширяет место на ней 
`lvextend` - команда увеличивает существующий логический том и ФС на нем 

Для перемещения данных используется команда `pvmove`:
- `sudo pvmove /dev/vdb1 /dev/vdb2`

*Такую команду лучше запускать в сессии терминального мультиплексора, например `screen` или `tmux`, чтобы не потерять данные, если сетевое соединение нарушится* 
## ***Перенос дисков между разными серверами***

*Если материнская плата сервера вышла из строя, а данные на дисках остались целыми, можно перенести диски на другой сервер и восстановить важную информацию. Для этого — установить LVM на новом сервере и активировать группу томов.*

***Чтобы воспроизвести этот сценарий, можно отмонтировать логический том, удалить пакет `lvm2` и перезагрузить сервер:***
- `sudo umount /mnt/data`
- `sudo apt remove lvm2 --purge --autoremove` 

***Для восстановления из данной ситуации после перезагрузки нужно установить пакет `lvm2` и активировать группу томов:***
- `sudo apt install lvm2` 
- `sudo vgs` 
- `sudo vgchange -ay` 
- `sudo mount /dev/data/lv /mnt/data`
## ***Траблшутинг***

***При повреждении метаданных, потере физических томов и других сбоях могут помочь специальные команды для траблшутинга и восстановления данных в LVM. Рассмотрим основные:***
1. `pvscan` - сканирует доступные устройства на наличие физических томов LVM, а также обновляет информацию о них в системе. Ключ `-d` - выводит дополнительные данные о найденных физических томах 
2. `vgscan` - проверяет систему на группы томов LVM, обновляет инфу о них 
3. `lvscan` - ищет в системе логические тома и актуализирует о них информацию 
4. `vgcfgrestore` с помощью данной команды можно попытаться восстановить предыдущие метаданные группы томов из резервной копии, если они повреждены или удалены. `f <file>` - ключ указывает файл с резервной копией метаданных. `<vg_name>` - ключ, имя группы томов для восстановления 
5. `pvcreate --uuid <UUID> --restorefile <file>` - восстанавливает метаданные физического тома LVM из резервной копии, указанный в параметре restorefile. `uuid <UUID>` - ключ, указывает резервный UUID для восстановления. `restorefile <file>` - ключ, резервный файл 
6. `lvchange -ay` - если логический том не отмечен как активный, данный ключ поможет восстановить его активность 
7. `lvremove --force <lv_path>` - удаляет логический том, который не удается восстановить, а ключ `force` позволяет удалить даже поврежденные тома 
8. `pvmove --abort` - если перемещение данных между физическими томами LVM прервалось или возникла другая проблема, этот ключ поможет отменить операцию 

***Конкретная ситуация*** 

*Допустим, у нас есть система с LVM. Она состоит из физических томов `/dev/sda1` и `/dev/sdb1`, объединённых в группу томов `vg01`, а также логических томов `lv_data` и `lv_backup`.*

*По какой-то причине жёсткий диск `/dev/sda1` с важными данными вышел из строя, и система перестала видеть этот физический том.*

***Вот какие шаги нам следует предпринять:***

- ***Обнаружение проблемы.***
1. Запускаем команду `pvdisplay`, чтобы посмотреть список физических томов. Видим, что `/dev/sda1` отсутствует.
2. Проверяем системные журналы на сообщения об ошибках или потере доступа к диску.
- ***Исправление проблемы.***
1. Подключаем новый жёсткий диск `/dev/sdc1` к системе.
2. Запускаем `pvcreate /dev/sdc1`, чтобы создать новый физический том на добавленном диске.
3. Через `vgextend vg01 /dev/sdc1` добавляем новый физический том в группу томов `vg01`.
4. Создаём новую `temporary logical volume lv_temp` для переноса данных со сломанного диска: `lvcreate -n lv_temp -L 10G vg01`.
5. Копируем информацию на новый диск: `pvmove /dev/sda1 /dev/sdc1`.
6. Удаляем повреждённый физический том: `pvremove /dev/sda1`.
7. И удаляем временный: `logical volume lvremove /dev/vg01/lv_temp`
