***Система контроля версий (англ. Version Control System, или VCS)*** - это ПО, которое помогает отслеживать изменения в программах, тестовых файлах, больших документах, веб-сайтах и так далее 
## ***Основные понятия***

***Репозиторий*** - место хранения данных и истории их изменений. По сути это папка в ФС с метаданными git 
- `.git` - служебная информация 
- `.gitignore` - списки файлов, для которых не нужно отслеживать историю изменений 

***Git*** - распределённая система контроля. Это означает, что выделенного сервера нет и у каждого пользователя хранится полная копия репозитория на момент синхронизации 

*Репозиторий может быть локальным и храниться на пк пользователя. Или же удаленным - на сервере или в облаке* 

***Коммит (от англ. сommit — «фиксировать, передать в работу»)*** - команда фиксирующая внесенные данные в историю 

***Форк (от англ. fork — «вилка»)*** - процедура копирования репозитория 

***Мёрдж (от англ. merge — «сливать, соединять»), или слияние*** - соединение двух разных историй изменений 

***Ветка (англ. branch — «ответвление»)*** - ветвление истории внутри одного репозитория без форка, то есть копирования

***Your branch is based on 'origin/main', but the upstream is gone*** - говорит о том, что в локальном репозитории создана ветка `main`, связанная с отсутствующей веткой `main` в облачном репозитории на GitHub

`origin` - название удаленного репозитория 

`git push` - отправить локальные изменения в облачный репозиторий

***Ошибка: ветки `shell-script` нет в облачном репозитории. Значит, нужно указать соответствие веток между локальным и облачным репозиторием. Вот так:***

- `git push --set-upstream origin shell-script`

***Или же можно задать настройку, чтобы Git всякий раз делал это автоматически:***

- `git config --local --add --bool push.autoSetupRemote true`

#### ***Pull Request — запрос на изменения***

***Pull request, или PR*** - создает запрос с предложением забрать (применить, влить) изменения из одной ветки в другую. Проще говоря, это когда мы предлагаем другим разработчикам принять наши изменения 

*При работе в команде важно давать исчерпывающее описание запросу: чем лучше оформлен PR, тем больше вероятность его принятия*

***Merge, или слияние*** - это процесс соединения историй изменений
