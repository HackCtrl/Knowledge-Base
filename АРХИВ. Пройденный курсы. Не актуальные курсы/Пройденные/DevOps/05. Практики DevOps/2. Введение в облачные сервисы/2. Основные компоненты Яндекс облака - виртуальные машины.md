## ***Знакомство с Яндекс Облаком***

#### ***Облачные сети***

Это фундамент в облаке, при регистрации создается автоматически. Для этой задачи используется **Yandex Virtual Private Cloud, или** **VPC** - виртуальное частное облако 

*VPC предназначен для управления облачными сетями, связи облачных ресурсов между собой и с интернетом. Он позволяет создавать сети и подсети в зонах доступности Яндекс облака и назначать облачным ресурсам внутренние и публичные IP-адреса* 

***VPC используется для выполнения сетевых настоек:***
- Создание подсети и подключение к ним облачных ресурсов 
- Обеспечение передачи информации между облачными ресурсами и соединение их с интернетом 
- Выстраивание статических маршрутов и управление трафиком с помощью групп безопасности 
- Направление трафика в интернет через NAT-шлюз

***Подсеть*** - это диапазон IP-адресов в облачной сети. Адреса из этого диапазона могут назначаться облачным ресурсам вроде виртуальных машин. При этом создавать подсети можно только в том каталоге, которому принадлежит облачная сеть
#### ***Что же такое `CIDR`?***

***`CIDR` (Classless Inter-Domain Routing)*** - это способ распределять диапазон IP-адресов и указывать, сколько именно адресов входит в конкретную сеть. Главная проблема, которую он решает - экономия IP-адресов
#### ***Из чего состоит CIDR?***

Префикс сети или маска сети. Раньше размер сети был жёстко задан (Класс `A, B, C`). `CIDR` убрал эти классы и ввёл понятие **префикса,** например `192.168.1.0/24`. В данном случае `/24` показывает количество единиц в маске сети

![[Снимок экрана — 2026-05-19 в 09.16.07.png]]

`/24` означает, что первые **24 бита** из 32 — это адрес сети, а оставшиеся 8 (получаются вычитнием из `32-24`), а `2^8=256` , показывают адреса для устройств (хостов) внутри сети

Также у подсетей должны быть непересекающиеся диапазоны IP-адресов в пределах одной облачной сети. Первые два адреса из любого диапазона выделяются под шлюз (`x.x.x.1`) и DNS-сервер (`x.x.x.2`).

Вся магия `CIDR` в возможности изменения размера сети при замене числа после слеша (которое как раз и называется префиксом)

***Примеры:***
- `192.168.1.0/28`. Сеть, в которой `32-28=4 бита`, которые используются для расчёта хостов. Получается `2^4=16` адресов, что подойдёт для маленькой домашней сети.
- `192.168.1.0/23.` В такой сети уже больше возможных адресов, и она подойдёт для крупного офиса. Получается `32 - 23 = 9 бит` для хостов и `2^9 = 512` адресов.

***Несколько шагов создания подсети:*** 
1. В списке облачных сетей выбирается имя ранее созданной сети, например `test-network`.
2. В разделе «Подсеть» используется кнопка «Создать».
3. Указывается имя подсети, например `test-subnet-1`.
4. Из выпадающего списка выбирается зона доступности — для первой подсети подходит любая.
5. В поле CIDR задаётся IP-адрес и маска подсети, например `10.10.0.0/24`.
6. После заполнения параметров используется кнопка «Создать подсеть».
#### ***Группа безопасности***

*Применяется для разграничения сетевого доступа в облаке. Они состоят из списка правил (rules)*

***Свойства группы безопасности (Default Security Group, или DSG):*** 
- Разрешает в новой сети весь сетевой трафик в обоих направлениях 
- Действует для трафика, который проходит через все подсети в сети, где она создана 
- Работает лишь в том случае, если на объект еще явно не назначена группа безопасности 
- Удаляется только вместе с удалением сети 

*Группы безопасности можно комбинировать - всего на один объект мы можем назначить до 5 групп* 

***Процесс создания группы безопасности включает следующие этапы:***
1. В консоли управления выбирается каталог, в котором будет создана группа.
2. В списке сервисов открывается **Virtual Private Cloud**.
3. На панели слева выбирается раздел «Группы безопасности». Здесь уже есть группа по умолчанию, но при необходимости можно добавить новую.
4. Для создания новой группы используется кнопка «Создать группу безопасности».
5. Указывается имя группы, например `test-security-group`.
6. В поле «Сеть» выбирается сеть, к которой будет привязана группа.
7. В блоке «Правила» задаются параметры управления трафиком:
    - на вкладках «Исходящий трафик» или «Входящий трафик» можно добавить новое правило;
    - в параметрах указывается диапазон портов, протокол и источник или назначение трафика:
        - вариант «CIDR» позволяет применить правило к диапазону IP-адресов;
        - вариант «Группа безопасности» задаёт взаимодействие для ресурсов, на которые назначена текущая или другая группа.
    - При необходимости добавляются дополнительные правила.

8. Завершение процесса выполняется через кнопку «Создать».
## ***Настройка доступа в интернет***

С NAT-шлюзом можно предоставлять ресурсам доступ в интернет, не назначать им публичные адреса - они получают адрес из отдельного диапазона таких IP-адресов 

***Шлюз*** - региональный ресурс, который есть во всех зонах доступности 

***Создание шлюза в VPC выполняется в следующей последовательности:***
1. В консоли управления выбирается каталог, где будет размещён шлюз.
2. В списке сервисов открывается **Virtual Private Cloud**, затем на панели слева раздел «Шлюзы».
3. Используется кнопка «Создать шлюз».
4. Задаётся имя, например `test-nat`. К имени предъявляются требования: длина от 3 до 63 символов; допускаются строчные латинские буквы, цифры и дефисы; первый символ — буква, последний — не дефис.
5. При необходимости добавляется описание.
6. Тип шлюза по умолчанию — NAT.
7. Изменения сохраняются через кнопку «Сохранить».
8. На панели слева открывается раздел «Таблицы маршрутизации».
9. Можно создать новую таблицу или использовать одну из существующих.
10. Для новой таблицы задаётся имя, например `test-route`.
11. Добавляется маршрут.
12. В настройках маршрута в поле **Next hop** указывается «Шлюз».
13. В поле «Шлюз» выбирается созданный NAT-шлюз, префикс назначения подставляется автоматически.
14. Параметры сохраняются.
## ***Работа с виртуальными машинами***

***Прерываемые виртуальные машины*** - могут быть принудительно остановлены в любой момент. Это происходит в двух случаях: если с момента запуска прошло 24 часа или же при нехватке ресурсов для запуска обычной ВМ в той же зоне доступности

***Зона доступности*** - эта инфраструктура внутри дата-центра, в котором размещается платформа Яндекс Облака. Ее ресурсы делятся на зональные - привязанные к зоне доступности, и глобальные - не привязанные к зоне, и каждая зона изолирована от аппаратных и программных сбоев в других 

***Для создания виртуальной машины нужно проследовать по шагам:***
1. В консоли управления выбрать каталог, где будете создавать ВМ, а в списке сервисов — Compute Cloud.
2. Нажать кнопку «Создать ресурс» → «Создать виртуальную машину».
3. В блоке «Базовые параметры»:
    a. Ввести имя ВМ, например `test-vm`. К нему есть требования: длина — от 3 до 63 символов, оно может включать строчные буквы латинского алфавита, цифры и дефисы, при этом первый символ должен быть буквой, а последний — не дефисом.

***FQDN от англ. Fully Qualified Domain Name*** — «полностью определённое доменное имя», иногда сокращается до «полное доменное имя»

b. Выбрать зону доступности, в которой будет находиться машина.
4. В поле «Выбор образа/загрузочного диска» выбирается Ubuntu последней доступной версии.
5. В поле «Диски и файловые хранилища» на вкладке «Диски» настройте загрузочный диск: задайте тип HDD и укажите размер — 20 Гб. Это опциональный шаг.
6. В блоке «Вычислительные ресурсы»:
    a. Выбрать поле «Своя конфигурация». b. Платформа оставляется по умолчанию. c. Задаются гарантированная доля, количество vCPU и объём RAM — например, 2 Гб. Ползунки vCPU и RAM влияют на стоимость, актуальные значения отображаются в правой части экрана. d. Если сделать ВМ прерываемой, это повлияет на итоговую стоимость и значительно снизит расходы.
7. В блоке «Сетевые настройки»:
    a. Указывается облачная сеть и подсеть, например `test-network/test-subnet-1`. b. В поле «Публичный адрес» выбирается вариант «Без адреса». c. Группу безопасности оставляем без изменений. d. При раскрытии блока «Дополнительно» в поле «Внутренний IPv4-адрес» есть опция «Автоматически», которая назначает случайный IP-адрес из пула доступных в выбранной подсети адресов
    При необходимости может быть добавлен дополнительный сетевой интерфейс; для каждой добавленной сетевой карты повторяются сетевые параметры. Допускается до восьми интерфейсов на одну ВМ.
8. В блоке «Доступ» указываются данные для доступа на ВМ:
    a. Опция «Доступ через OS Login» может быть отключена. b. В поле «Логин» указывается имя пользователя.
9. В поле «SSH-ключ» вставляется содержимое файла открытого ключа. Пара ключей для SSH-подключения должна быть подготовлена заранее.
10. При нажатии кнопки «Создать ВМ» машина появится в списке.
